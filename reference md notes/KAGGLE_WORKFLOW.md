# Kaggle CLI Complete Workflow Guide

This guide covers the entire end-to-end workflow for running machine learning tasks on Kaggle using the Kaggle Command Line Interface (CLI). 

While `run.py` attempts to automate this entire process for you locally, this document explains the underlying CLI commands being executed.

---

## 1. Authentication & Setup

Before using the CLI, you must authenticate your machine with your Kaggle account.

1. Go to **Kaggle.com** → **Settings** → **Create New Token**.
2. This downloads a `kaggle.json` file containing your username and API key.
3. Place this file in the designated global Kaggle directory:
   - **Windows:** `C:\Users\<YourUsername>\.kaggle\kaggle.json`
   - **Mac/Linux:** `~/.kaggle/kaggle.json`
4. *(Mac/Linux only)* Secure the file permissions:
   ```bash
   chmod 600 ~/.kaggle/kaggle.json
   ```

---

## 2. Dataset Management

If your script relies on local data (like CSV files or images), you must upload them to Kaggle as a "Dataset" before your code can access them.

### A. Initialize a New Dataset
Create a folder containing your data files, then generate a metadata template:
```bash
kaggle datasets init -p /path/to/data/folder
```
This generates a `dataset-metadata.json` file. Open it and configure the `"id"` (e.g., `"yourusername/my-dataset"`) and `"title"`.

### B. Create the Dataset on Kaggle (First Time Only)
Upload the initial version to Kaggle:
```bash
kaggle datasets create -p /path/to/data/folder
```
> ⚠️ **Known Kaggle CLI Bug:** `kaggle datasets create` returns exit code `0` (success) even when it fails because the dataset already exists. Always inspect the output text, not just the exit code.

### C. Update / Push a New Version (Most Common)
When you change the CSV or add new files, always use `version`, **not** `create`:
```bash
kaggle datasets version -p /path/to/data/folder -m "Updated: new_filename.csv"
```
This is the **correct command** for updating an existing dataset. `run.py` calls this automatically.

### D. List All Files in a Dataset
Verify what Kaggle actually has stored (useful for debugging path issues):
```bash
kaggle datasets files yourusername/dataset-slug
```
Example:
```bash
kaggle datasets files hareeshks/vc-stock-data
```

### E. List All Your Datasets
```bash
kaggle datasets list --mine
```

## 3. Kernel (Code) Execution

Kaggle refers to scripts and notebooks as "Kernels". You push your local script to Kaggle, which then allocates a cloud container (optionally with a GPU) to run it.

### A. Initialize Kernel Metadata
In the folder containing your Python script, generate the kernel metadata template:
```bash
kaggle kernels init -p /path/to/script/folder
```
This generates `kernel-metadata.json`. You must edit this file to configure your run. Crucial fields include:
- `"id"`: `yourusername/kernel-slug`
- `"title"`: `Kernel Title` (must match the slug)
- `"code_file"`: `your_script.py`
- `"language"`: `python`
- `"kernel_type"`: `script` (or `notebook` for Jupyter)
- `"enable_gpu"`: `true`
- `"dataset_sources"`: `["yourusername/my-dataset"]` (Attaches the dataset you made in Step 2!)

### B. Push and Execute the Kernel
Pushing the kernel implicitly commands Kaggle's servers to begin executing the script from top to bottom.
```bash
kaggle kernels push -p /path/to/script/folder
```
*(⚠️ **Pro-tip**: Only push a staging folder containing your script and the metadata file. If you push your entire project directory, Kaggle will try to upload your `venv` and massive local `.PTH` models, causing the push to silently fail!)*

---

## 4. Monitoring the Run

You can poll the live execution status (`running`, `complete`, `error`, `queued`) from your terminal without opening the website:
```bash
kaggle kernels status yourusername/kernel-slug
```

---

## 5. Retrieving Results

Any files generated by your script inside the cloud container's `/kaggle/working/` directory can be downloaded back to your local PC once the kernel completes.

```bash
kaggle kernels output yourusername/kernel-slug -p /local/save/path
```
This will download all your saved `.pth` models, `.xlsx` result sheets, and logs straight back to your computer.

## 6. Troubleshooting

### Problem: "Data file not found" / mock data fallback
Kaggle script falls back to generated mock data instead of your CSV.

**Diagnosis steps:**
```bash
# 1. Check what files Kaggle actually has for your dataset
kaggle datasets files hareeshks/vc-stock-data

# 2. Confirm your local file matches what you expect
ls data/

# 3. Check the run.py hash cache to see what filename it last uploaded
Get-Content .last_dataset_hash         # (PowerShell)
cat .last_dataset_hash                 # (bash/Mac)
```

**Fix:**
```bash
# Force a fresh upload by clearing the hash cache
Remove-Item .last_dataset_hash -Force  # PowerShell
rm .last_dataset_hash                  # bash/Mac

# Then run the launcher again — it will detect no cache and upload
python run.py
```

**Root causes found in this project:**
1. `run.py` was calling `kaggle datasets create` (which silently fails if dataset exists) instead of `version`.
2. The MD5 hash cache (`filename:hash`) was matching even though Kaggle had an old file because the filename changed but file contents didn't.

---

### Problem: Dataset slug updates the file but Kaggle kernel still references old data
After uploading a new version to your Kaggle dataset, the running kernel may not see the new file. This is expected — the kernel must be re-pushed after the dataset version is updated.

```bash
# Force re-push of kernel after dataset version update
kaggle kernels push -p /path/to/kernel/staging
```

---

### Problem: `run.py` skips dataset upload even though the file/filename changed
The cache file (`.last_dataset_hash`) stores `filename:md5_hash`. If the cache matches the current file, `run.py` skips upload. Clear it:

```bash
# PowerShell
Remove-Item .last_dataset_hash -Force

# bash
rm .last_dataset_hash
```

---

### Problem: Kaggle kernel shows `exit_code: 1` but no visible error
Enable verbose logging:
```bash
kaggle kernels status yourusername/kernel-slug
```
Then go to **Kaggle.com → Your Kernel → Logs tab** for the full output.

---

### Problem: `UnicodeEncodeError` / emoji crash during kernel push
Kaggle CLI on Windows crashes on non-ASCII characters (emojis like 🚨) when using the `cp1252` codec.

**Fix (already in `run.py`):** Strip non-ASCII characters from the script before pushing:
```python
safe_content = new_content.encode("ascii", "ignore").decode("ascii")
```

## 7. Key Paths on Kaggle's Server

| Purpose | Kaggle Path |
|---|---|
| Your uploaded dataset files | `/kaggle/input/<dataset-slug>/<filename>` |
| Script working directory | `/kaggle/working/` |
| Output files for download | `/kaggle/working/` |

**Example for this project:**
- Dataset: `/kaggle/input/vc-stock-data/aapl_stock_sentiment_old_dataset_2016_2024.csv`
- Script: `/kaggle/src/script.py`

The `glob.glob("/kaggle/input/**/*.csv", recursive=True)` pattern in the script automatically finds the CSV regardless of subdirectory layout.
